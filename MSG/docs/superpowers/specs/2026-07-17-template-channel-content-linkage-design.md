# 模板正式渠道与内容编辑联动设计

## 背景

消息模板抽屉当前虽然允许选择“站内信”和“Push”正式渠道，但内容编辑区固定显示两套编辑器，保存时也固定校验两套内容，预览页默认展示全部渠道。这会让操作者误以为未选择的渠道仍需填写并会参与发布。

## 目标

让正式渠道成为模板内容编辑、预览、测试发送、保存校验和发布范围的唯一依据，使操作者只处理本次模板实际启用的渠道。

## 方案比较

### 方案 A：按渠道动态显示编辑模块（采用）

- 选择站内信时显示站内信编辑器；选择 Push 时显示 App Push 编辑器。
- 只选一个渠道时编辑器占满整行；同时选择时各占半行。
- 取消渠道只隐藏对应模块并保留本次抽屉会话中的输入，重新选择后恢复。

优点是界面与发布范围直接对应，认知成本最低，也与人工临时消息的现有编辑方式一致。缺点是切换渠道时页面布局会变化，但变化由操作者主动选择触发，预期明确。

### 方案 B：始终显示两套编辑器，未选渠道置灰

优点是布局稳定，操作者能看到所有字段。缺点是未选渠道仍占据大量空间，并容易让人误认为需要补齐，因此不采用。

### 方案 C：通过渠道页签切换编辑器

优点是单次只显示一套内容。缺点是同时选择两个渠道时不便对照编辑，也增加页签状态与渠道选择同步的复杂度，因此不采用。

## 交互设计

### 正式渠道

- 正式渠道至少选择一个；当操作者尝试取消最后一个渠道时，保留当前选择并提示“请至少保留一个正式渠道”。
- 可选项为“站内信”和“Push”。
- 渠道变化立即作用于内容编辑区和预览区，无需额外确认。

### 内容编辑

- 选择“站内信”：显示“站内信（Web + App 共用）”编辑模块。
- 选择“Push”：显示“App Push”编辑模块。
- 同时选择：两套模块左右各占 12/24 栅格。
- 只选择一个：对应模块占满 24/24 栅格。
- 取消某渠道时不清空已输入内容；在当前抽屉会话中重新勾选后恢复。
- 未选渠道的暂存内容不参与校验、测试发送、预览和正式发布。

### 预览与测试发送

- “双端预览”按正式渠道过滤：站内信展示 Web 与 App 站内信预览，Push 展示 App Push 预览。
- 测试发送自动使用当前正式渠道，不允许发送到未选渠道。

## 数据与校验

- `channels` 是渠道范围的唯一事实来源。
- 保存草稿和提交外部机翻前必须校验 `channels.length > 0`。
- 站内信字段仅在选择站内信时校验标题、摘要、正文和 Markdown 链接安全性。
- Push 字段仅在选择 Push 时校验标题和正文。
- 内容对象继续保留两种渠道的数据结构，以支持取消后重新选择时恢复输入；下游只根据 `channels` 处理启用渠道。

## 影响范围

- 修改 `src/pages/templates/TemplateEditorDrawer.tsx`：增加受控渠道状态、动态编辑模块、按渠道校验和预览过滤。
- 新增 `src/pages/templates/TemplateEditorDrawer.channel-linkage.test.tsx`：覆盖单渠道显隐、单渠道校验、重新选择恢复和预览过滤。
- 不修改消息模板的数据类型、存储结构、已审核模板只读规则和外部翻译流程。

## 验收标准

1. 仅选择站内信时，不显示 Push 编辑器和 Push 预览，允许在站内信内容完整后保存或提交。
2. 仅选择 Push 时，不显示站内信编辑器和站内信预览，允许在 Push 内容完整后保存或提交。
3. 同时选择两个渠道时，两套编辑器同时显示且各占半行，两套内容均需完整。
4. 不允许取消全部正式渠道。
5. 取消渠道再重新选择，之前输入的该渠道内容仍然存在。
6. 测试发送仅包含当前选择的正式渠道。
