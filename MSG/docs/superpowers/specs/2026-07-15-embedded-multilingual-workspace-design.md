# 多语言生产、进度与小语种专审设计

## 1. 背景

当前消息中心有三个多语言生产入口：

1. 消息模板；
2. 人工任务中的临时消息；
3. 事件通知规则的内容版本。

三个入口都需要选择目标语言、调用外部机器翻译、查看进度、人工修改译文、逐语言确认、失败重翻和发布门禁。现有实现以模板 ID 为翻译批次的主要归属，导致临时消息需要创建隐藏模板承载翻译任务，事件规则又维护另一套内容版本状态，进度和操作容易分散。

## 2. 设计结论

- 在三个内容生产入口内嵌同一个“多语言工作区”。
- 新增独立“多语言审核”导航，只承接配置为需要专审的语言，不承担普通语言生产。
- 系统配置新增“语言审核策略”，逐语言配置是否需要专审、审核组、审核人数和时限。
- 原页面和多语言审核页面操作同一份翻译批次与语言子任务数据。
- 外部机翻采用混合交互：用户可以留在页面等待并当场校对，也可以离开，完成后通过通知和审核待办继续处理。
- 普通语言允许创建人在原页面修改并确认；专审语言在原页面只展示进度，必须由配置的语言审核组修改和确认。
- 消息模板、人工消息任务和事件规则内容版本列表都直接展示多语言流程、完成比例和未完成语言。

## 3. 目标

1. 三个入口的多语言能力、字段、状态和操作完全一致。
2. 用户不需要理解回调、主动查询或供应商任务协议即可完成工作。
3. 机器翻译快速返回时，用户可以在当前页面立即校对。
4. 翻译耗时较长时，用户可以安全离开并从原业务对象或多语言审核页面恢复进度。
5. 人工修改稿、机器原始译文、审核结论和源文案版本均可追溯。
6. 源文案发生变化时只重翻受影响字段，同时保证旧审核结果不会错误复用。

## 4. 非目标

- 不建设通用翻译供应商管理平台。
- 不支持翻译记忆库、术语库运营后台和机器翻译质量模型训练。
- 不在一期提供跨批次的翻译成本结算。
- 不把普通语言校对迁移到独立审核页面。

## 5. 用户体验

### 5.1 发起流程

```text
编辑源文案
└─选择目标语言
   └─生成机器翻译
      ├─快速返回→ 当前页面自动展开校对区
      ├─处理中→ 显示进度，用户可以离开
      ├─普通语言→ 当前页面校对并确认
      ├─专审语言→ 自动进入多语言审核
      └─失败→ 自动重试；仍失败时显示单语言重翻入口
```

操作按钮统一使用：

- `生成机器翻译`
- `保存修改`
- `重新机翻`
- `确认当前语言`
- `确认并查看下一种`
- `前往专审`

### 5.2 多语言工作区

工作区由四部分组成：

1. 摘要区：默认语言、目标语言、整体进度、业务对象和内容版本。
2. 语言列表：每种语言的当前状态、更新时间、审核人和可用操作。
3. 同屏校对区：普通语言左侧显示源文案，右侧可编辑机器译文；专审语言只显示状态和跳转入口。
4. 门禁区：展示距离提交业务审核还缺少哪些语言或检查项。

默认隐藏外部任务 ID、回调结果和查询次数。用户展开“技术详情”后才显示这些排障字段。

### 5.3 当场校对

普通语言机器翻译返回后不要求跳转页面。当前业务页面自动展开第一个待处理语言，操作者可以直接修改标题、摘要、正文、风险提示、按钮文案和 Push 文案。配置为专审的语言不能在原业务页面修改或确认，只能查看状态并前往多语言审核页面。

系统同时保留：

- `machine_output`：外部服务原始返回，不允许覆盖；
- `human_draft`：人工正在修改的草稿；
- `approved_output`：最终确认译文；
- `review_history`：每次保存、重翻、确认和驳回记录。

### 5.4 离开与恢复

- 翻译任务在后台继续执行，关闭页面不会取消任务。
- 机翻完成、失败或被驳回时发送后台站内通知。
- 再次打开原模板、临时任务或事件规则内容版本时恢复同一批次、选中语言和人工草稿。
- 多语言审核页面点击待办时打开专审校对区，并保留返回原业务对象的入口。

### 5.5 列表中的多语言流程

消息模板、人工消息任务和事件规则内容版本列表统一增加“多语言流程”列。列表单元格必须同时回答三个问题：完成多少、当前在哪一步、还差哪些语言。

```text
3/6 已通过 · 50%
源文案 ✓ → 机翻 ✓ → 普通确认 ✓ → 小语种专审 2/3 → 完成
未完成：日语·待专审 / 土耳其语·专审中 / 俄语·翻译失败
```

流程固定为：

```text
源文案完成 → 生成机器翻译 → 普通语言确认 → 小语种专项审核 → 全部语言通过
```

语言不超过三种时全部列出。未完成语言超过三种时，列表显示阻断优先级最高的三种和“其余 N 种”；点击后打开右侧抽屉查看全部语言、状态、处理人、更新时间和下一步操作。

未完成语言是所有状态不为`已通过`且未取消的目标语言。聚合阻断优先级为：`翻译失败 > 源文案已变更 > 审核驳回 > 待小语种专审 > 待普通确认 > 机器翻译中 > 全部通过`。

## 6. 三个入口的接入规则

### 6.1 消息模板

- 工作区归属对象为模板版本。
- 全部语言确认后，模板版本才能提交业务审核。
- 已发布模板不允许原地修改；编辑会创建新模板版本和新翻译批次。
- 模板列表展示多语言流程、未完成语言和专审阻断数量。

### 6.2 临时消息

- 工作区直接归属人工任务的内容版本，不再通过隐藏模板表达业务归属。
- 创建机翻批次时自动保存人工任务草稿，确保离开页面后可恢复。
- 全部语言确认后，人工任务才能进入业务审核。
- 人工任务列表展示多语言流程；专审语言可以直接跳转到多语言审核。

### 6.3 事件通知规则内容版本

- 工作区归属规则内容版本，不影响长期规则的启停状态。
- 新内容版本完成翻译、语言确认和业务审核后才能切换为当前生效版本。
- 新版本处理期间，旧内容版本继续服务事件触发。
- 内容版本列表展示多语言流程，不改变长期规则的启停状态。

## 7. 独立多语言审核

新增独立“多语言审核”页面，只展示语言审核策略配置为需要专审的语言，统一接收三个来源：

| 字段 | 说明 |
|---|---|
| 来源 | 消息模板、临时消息、事件规则版本 |
| 业务对象 | 对象名称与可点击 ID |
| 内容版本 | 翻译绑定的不可变版本 |
| 目标语言 | 当前语言子任务 |
| 风险 | 普通、重要、紧急等 |
| 状态 | 待专审、专审中、已通过、审核驳回、需重翻 |
| 提交人 | 发起机器翻译的人 |
| 当前处理人 | 正在校对或最终复核的人 |
| 等待时长 | 从机翻完成开始计算 |
| 操作 | 开始专审、保存修改、通过、驳回重翻、查看源对象 |

多语言审核页面不复制另一份译文。它通过语言子任务 ID 打开同一份翻译数据，展示源文案、机器原始译文、专审修改稿、变量检查和完整审核轨迹。

## 8. 状态模型

### 8.1 翻译批次

```text
未提交 → 机翻处理中 → 普通语言确认中 → 小语种专审中 → 全部语言通过
                    ├→ 部分失败
                    ├→ 审核被驳回
                    └→ 已取消
```

### 8.2 语言子任务

```text
排队中 → 翻译中
             ├→ 待普通确认 → 修改中 → 已通过
             ├→ 待小语种专审 → 专审中 → 已通过
             ├→ 翻译失败 → 重新机翻
             └→ 审核驳回 → 重新机翻
```

语言是否进入小语种专审由“语言审核策略”决定，与消息业务审核的风险路由分别判断。专审语言只能由对应审核组处理。

## 9. 源文案修改与差异重翻

翻译批次绑定`subject_type + subject_id + content_version + source_hash`。每个可翻译字段单独保存字段哈希。

源文案修改后：

1. 系统比较标题、摘要、正文、风险提示、按钮文案和 Push 字段的哈希。
2. 只将发生变化的字段提交机器翻译。
3. 受影响字段的人工确认结果失效，状态回到`待人工确认`。
4. 未变化字段继续保留人工修改稿和确认结果。
5. 变量集合变化、变量被删除或变量结构不合法时，整种语言强制重新确认。
6. 已发布或已生效内容不原地覆盖，而是创建新内容版本。

工作区必须明确展示“哪些字段发生变化、哪些语言需要重新确认”，不能静默失效。

## 10. 统一数据模型

### 10.1 翻译归属

| 字段 | 类型 | 说明 |
|---|---|---|
| `subject_type` | enum | `template_version`、`manual_task_content`、`rule_content_version` |
| `subject_id` | string | 业务对象 ID |
| `subject_name` | string | 列表和通知展示名称 |
| `content_version` | string | 不可变内容版本 |
| `source_locale` | string | 默认语言 |
| `source_hash` | string | 整体源文案哈希 |
| `source_field_hashes` | object | 各可翻译字段哈希 |
| `risk_level` | enum | 决定是否需要他人复核 |
| `return_path` | string | 返回原业务对象的内部路由 |

### 10.2 翻译批次

翻译批次继续保存批次 ID、目标语言、整体状态、创建人、创建时间、更新时间和语言子任务。`template_id`降级为模板来源时的兼容字段，不再作为所有翻译任务的必填归属。

### 10.3 语言子任务

新增或明确以下字段：

- `machine_output`
- `human_draft`
- `approved_output`
- `changed_fields`
- `reviewer`
- `reviewed_at`
- `review_required`
- `special_review_required`
- `review_group`
- `external_task_id`
- `attempt_no`
- `error_code`
- `error_message`

### 10.4 语言审核策略

| 字段 | 说明 |
|---|---|
| `locale_code` | 语言编码，如`tr-TR`、`ru-RU` |
| `locale_name` | 后台展示名称 |
| `special_review_required` | 是否必须进入专审 |
| `review_group` | 可处理该语言的审核组 |
| `reviewer_count` | 单人或双人专审 |
| `allow_submitter_review` | 是否允许机翻提交人专审，默认关闭 |
| `review_sla_hours` | 审核时限 |
| `timeout_action` | 提醒、升级或阻断发布 |
| `status` | 启用或停用 |

## 11. 权限规则

- 普通语言由原业务页面的内容操作者修改并确认。
- 专审语言只能由语言审核策略指定的审核组修改和确认。
- 专审语言的最终审核人默认不能与机翻提交人相同。
- 消息风险对应的业务审核继续由审核与发布模块处理，不与小语种专审合并。
- 单语言重翻、人工修改、确认、驳回和源文案失效均写入审计日志。

## 12. 异常处理

- 外部接口超时：保持后台处理，页面显示最后更新时间并允许主动刷新。
- 单语言失败：不影响其他语言，允许只重试失败语言。
- 回调重复或乱序：使用外部任务 ID、尝试次数和状态优先级幂等合并。
- 人工修改未保存：离开前提示；已保存草稿可跨页面恢复。
- 源文案在审核中变化：阻止旧译文确认，展示差异并要求重新机翻变化字段。
- 业务对象被取消：翻译批次进入已取消，保留审计但不再接受修改。

## 13. 前端组件边界

`MultilingualWorkspace`负责原页面的翻译生产、普通语言校对、整体流程和专审进度，不负责模板发布、任务受众、发送策略或规则启停。它通过以下输入工作：

- 业务对象身份与返回路由；
- 内容版本、源语言内容、目标语言；
- 风险等级和权限；
- 翻译批次及操作回调。

三个业务页面负责保存自身草稿和处理业务审核提交；工作区只返回翻译门禁是否通过。

`MultilingualReviewPage`只处理专审语言队列；`MultilingualProgressCell`在三个业务列表中统一展示流程、比例和未完成语言；`MultilingualProgressDrawer`展示全部语言详情但不复制完整编辑能力。

## 14. 验收标准

1. 三个入口均展示同样的多语言工作区和操作名称。
2. 机器翻译快速返回后无需跳页即可修改译文。
3. 离开并重新进入后可恢复批次、进度和人工修改草稿。
4. 独立多语言审核页面只接收配置为专审的语言，并能返回三个来源的原业务对象。
5. 单语言失败只重试该语言。
6. 源文案修改后只重翻变化字段，受影响确认结果明确失效。
7. 机器原始译文、人工草稿和最终确认译文分别保留。
8. 普通语言允许原页面当场确认；专审语言只能由配置的语言审核组确认。
9. 全部目标语言通过前，模板、临时任务或规则内容版本不能进入后续业务发布流程。
10. 三个业务列表直接展示多语言流程、完成比例和未完成语言。
11. 系统配置可以逐语言启停专审、配置审核组、审核人数、审核时限和超时动作。
