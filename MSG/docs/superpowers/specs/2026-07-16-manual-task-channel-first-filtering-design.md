# 人工消息任务发送渠道前置筛选设计

## 背景

当前人工消息任务在第 1 步选择内容，在第 3 步才选择正式发送渠道。操作者选择内容时无法提前判断模板是否支持目标渠道；临时消息也会固定展示并校验站内信和 App Push 两套内容，造成无效填写和流程返工。

## 目标

- 在第 1 步“内容与多语言”先确定发送渠道。
- 使用所选渠道筛选可用模板、临时内容编辑器、翻译内容和预览。
- 第 3 步不再重复选择渠道，只展示第 1 步已经确定的结果。
- 任务保存、提交和预览始终使用同一组渠道。

## 方案选择

考虑过三种方式：

1. **前置选择并按覆盖关系筛选（采用）**：模板覆盖任务所选全部渠道即可使用，双渠道模板可以用于单渠道任务。复用率高，同时不会出现缺少渠道内容。
2. **仅前置选择、不筛选模板（不采用）**：改动较小，但仍可能选择缺少目标渠道内容的模板，不能解决返工问题。
3. **按渠道完全一致筛选（不采用）**：规则最严格，但双渠道模板无法用于单渠道任务，会制造不必要的重复模板。

## 渠道选择规则

- 可选渠道为“站内信（Web + App）”和“App Push”。
- 至少选择一个渠道，未选择时不能继续下一步。
- 默认同时选择两个渠道；复制或再次编辑任务时使用任务原有渠道。
- 双渠道模板可以用于单渠道任务。
- 单渠道模板不能用于双渠道任务。
- 模板可用条件为：模板支持的渠道必须覆盖任务选择的全部渠道。

## 第 1 步界面

“内容与多语言”的展示顺序调整为：

1. 任务基础信息。
2. 发送渠道。
3. 内容来源。
4. 模板选择或临时消息编辑。
5. 多语言生产与审核进度。
6. 当前渠道内容预览。

渠道切换后的行为：

- 重新计算模板候选列表。
- 当前模板不再支持全部已选渠道时，自动切换到第一个可用模板。
- 没有可用模板时展示空状态，阻止继续下一步，并允许改用临时消息。
- 临时消息仅展示所选渠道对应的编辑区。
- 单渠道任务不校验、翻译或预览未选渠道的内容。

## 多语言规则

- 机翻任务只提交所选渠道的源文案。
- 站内信源文案包括标题、摘要、正文、风险提示和操作文案。
- App Push 源文案包括标题和正文；图片、设备平台、优先级、折叠键和 Deep Link 作为渠道配置保留，不进入正文机翻。
- 已生成机翻批次后修改渠道会使原批次与当前内容范围不一致。页面保留现有批次记录，但要求重新创建机翻批次后才能提交业务审核。

## 第 3 步界面

- 移除“正式渠道”的复选框。
- 使用只读标签展示已选渠道。
- 保留发送模式、计划时间、任务时区、有效期、频控与安静时段等发送策略。
- 仅选择站内信时不展示 App Push 正式发送检查。
- 选择 App Push 时继续展示 App Push 正式发送检查。

## 数据与校验

- `submission.channels` 直接使用第 1 步表单中的渠道值。
- 保存草稿、预览、提交和复制任务共用该值。
- 第 1 步校验：至少一个渠道；模板覆盖全部所选渠道；临时消息完成全部所选渠道必填文案。
- 最终提交仍再次校验至少一个渠道和内容完整性，避免绕过步骤校验。

## 测试范围

- 第 1 步显示发送渠道，第 3 步不显示可编辑渠道控件。
- 双渠道模板可用于单渠道任务，单渠道模板不能用于双渠道任务。
- 切换渠道会更新模板列表并修正当前模板。
- 仅站内信时只展示和校验站内信内容。
- 仅 App Push 时只展示和校验 Push 内容。
- 双渠道时展示并校验两套内容。
- 第 3 步按渠道决定是否展示 App Push 正式发送检查。
- 保存与提交结果使用第 1 步选择的渠道。

## 非目标

- 不改变模板自身支持渠道的配置方式。
- 不改变站内信在 Web 和 App 共用内容的规则。
- 不调整消息分类默认性质、受众、排期和频控规则。
