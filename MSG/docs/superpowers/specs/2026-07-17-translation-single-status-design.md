# 多语言单状态模型设计

## 目标

将当前混合了外部机翻过程、人工审核过程和小语种路由的多语言状态收敛为一条业务状态链。操作者只需要判断是否已有译文、是否需要审核、是否已经通过，不再看到外部任务的排队和处理中等技术过渡态。

## 状态模型

目标语言条目和翻译批次统一使用以下三个业务状态：

```text
无结果 → 翻译返回待审核 → 已通过
```

| 状态 | 定义 | 主要操作 |
| --- | --- | --- |
| 无结果 | 当前源文案没有可用于人工审核的机器译文 | 发起或重新发起机翻 |
| 翻译返回待审核 | 已存在可查看、可修订的机器译文，尚未完成人工审核 | 普通语言当场审核；专项语言前往多语言审核 |
| 已通过 | 当前译文已通过对应的人工审核 | 查看最终结果 |

## 旧状态映射

| 旧状态 | 新状态或处理方式 |
| --- | --- |
| 未提交、排队中、翻译中、翻译失败 | 无结果 |
| 源文案已变更 | 无结果，并单独展示“源文案已变更”警告 |
| 待普通确认、修改中、待人工审核 | 翻译返回待审核 |
| 待小语种专审、专审中 | 翻译返回待审核，并展示“专项审核”标签 |
| 审核驳回 | 翻译返回待审核，并单独展示最近驳回原因 |
| 审核通过、已通过 | 已通过 |
| 已取消 | 不计入当前有效语言集合；取消信息仅保留在审计记录中 |

外部任务 ID、请求时间、回调时间、错误码和错误信息继续作为技术追踪字段保存，但不再成为业务状态。

## 批次聚合规则

翻译批次同样只显示一个状态：

1. 任一有效目标语言为“无结果”，批次为“无结果”。
2. 全部语言均有结果，但至少一个语言未通过，批次为“翻译返回待审核”。
3. 全部有效目标语言均为“已通过”，批次为“已通过”。

批次旁继续展示完成比例和缺少结果的语言，例如：

```text
状态：无结果
审核通过：3/6
缺少结果：ru-RU
```

## 审核和小语种路由

- 普通语言处于“翻译返回待审核”时，可以在当前多语言结果面板中校对并通过。
- 配置为小语种专项审核的语言同样显示“翻译返回待审核”，但附加“专项审核”标签，只能跳转独立多语言审核页处理。
- 审核驳回不创建新状态。条目保持“翻译返回待审核”，展示驳回原因，并允许修订后再次审核。
- 发布门禁保持不变：全部有效目标语言必须为“已通过”。

## 前端调整

- 任务列表、模板列表和事件内容版本列表统一展示三状态、通过比例和缺少结果的语言。
- 多语言进度抽屉删除“机器翻译中、普通确认、小语种专审中”等阶段步骤，只展示批次状态、比例和逐语言结果。
- 逐语言行只显示三状态；专项审核以标签表达，不作为状态。
- “无结果”显示重新机翻操作；“翻译返回待审核”根据审核类型显示普通审核或专项审核入口；“已通过”只允许查看。
- 错误、源文案变更和驳回原因使用提示信息展示，不扩展状态枚举。

## 数据与兼容

- 前端领域类型将目标语言和批次状态都收敛为三值枚举。
- Mock 数据和本地存储读取时执行旧状态迁移，避免浏览器已有数据导致页面异常。
- 旧状态只用于迁移映射，不再由任何新操作写入。

## 验收标准

1. 所有业务页面只显示“无结果、翻译返回待审核、已通过”三种多语言状态。
2. 小语种专项审核仍可识别和跳转，但不产生额外状态。
3. 审核驳回后状态回到“翻译返回待审核”，且驳回原因可见。
4. 源文案变化和机翻失败不会产生新状态，分别以警告和错误详情表达。
5. 只有全部目标语言为“已通过”时，模板、临时消息或事件内容版本才能进入后续业务审核与发布。
6. 旧的本地模拟数据可以无损映射到新状态模型。
