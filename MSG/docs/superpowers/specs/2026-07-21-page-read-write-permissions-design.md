# 页面读写权限设计

## 背景与目标

现有人员权限使用“创建内容、提交审核、模板变量维护、业务审核、风控审核”等能力项，无法直接回答操作者能否进入某个页面、能否在页面内执行修改。此次改为按后台页面和系统配置子 Tab 配置读、写权限，并由同一份权限数据同时控制导航、直接 URL 访问和页面写操作。

多语言审核不使用普通页面读写权限。它继续按照具体语言分配审核权限，操作者至少被授权一种语言后才可进入多语言审核页面，并且只能查看和审核获授权的语言。

## 权限模型

每位操作者保存 `pagePermissions`，按权限资源记录：

```ts
type PageAccess = {
  read: boolean;
  write: boolean;
};

type PagePermissionMap = Record<PagePermissionKey, PageAccess>;
```

规则如下：

- 写权限必须包含读权限。开启写权限时自动开启读权限；关闭读权限时同时关闭写权限。
- 超级管理员固定拥有全部页面的读写权限，不允许被降权。
- 人员权限配置只对超级管理员显示和开放。
- 非超级管理员的权限与岗位、团队无关，只按个人配置。
- 账号停用时保留权限数据，但不允许编辑权限。

## 权限资源

一级页面权限：

1. 工作台
2. 用户与受众
3. 人工消息任务
4. 人工消息模板
5. 模板变量
6. 事件通知规则
7. 事件消息模板
8. 事件目录
9. 触发记录
10. 审核中心
11. 发送记录
12. 数据分析

系统配置不设置额外的父级权限。只要操作者拥有任意一个系统配置子 Tab 的读权限，侧边栏就显示“系统配置”；进入后打开第一个可读 Tab。子 Tab 分别配置：

1. 消息分类
2. 跳转白名单
3. 语言审核策略
4. 测试账号
5. 审计日志

“人员权限”是超级管理员专用 Tab，不向普通操作者提供可配置的读写项；超级管理员固定可读写。

## 路由与继承

- `/` 与 `/dashboard` 使用“工作台”权限。
- `/tasks` 与 `/tasks/create` 使用“人工消息任务”权限；创建、再次编辑等操作要求写权限。
- `/templates?scope=manual` 使用“人工消息模板”权限。
- `/templates?scope=event` 使用“事件消息模板”权限。
- `/settings?tab=<key>` 使用对应系统配置子 Tab 的权限。
- `/multilingual-review` 依据语言授权判断，不读取普通页面权限。
- 其他页面按其导航 Tab 使用对应权限。

直接访问无读权限的 URL 时显示明确的“无权访问”页面，不自动跳转。系统配置入口本身是聚合入口：从侧边栏进入时跳到第一个可读子 Tab；直接访问一个无读权限的子 Tab 时显示“无权访问”。

## 导航行为

- 没有读权限的一级页面不显示在侧边栏。
- 一个导航分组下没有任何可读子页面时，整个分组隐藏。
- 至少有一个系统配置子 Tab 可读时显示“系统配置”。
- 至少拥有一种启用语言的审核授权时显示“多语言审核”。
- 面包屑和激活状态继续由当前路由决定，不单独维护权限副本。

## 页面内写操作

只读操作者可以查看详情、筛选、搜索、预览和导出。所有会改变业务数据或流程状态的操作要求写权限，包括但不限于：

- 新增、复制、编辑、删除、启用、停用；
- 提交、发布、取消、暂停、恢复、重试；
- 审核通过、驳回、修订；
- CSV 导入和其他数据写入操作。

没有写权限时保留相关按钮的位置，但按钮置灰，并通过提示说明“当前账号无写权限”。页面数据更新函数继续保留业务校验；界面权限只负责入口控制，不替代服务端权限校验。

多语言审核的写操作仍按语言授权判断，不继承“审核中心”或“语言审核策略”的写权限。

## 人员权限界面

人员列表继续展示操作者、账号状态和可审核语言。“已授权能力”改为“页面权限概览”，展示可读页面数量和可写页面数量。

编辑抽屉包含两块：

1. 页面权限矩阵：按“一级页面”和“系统配置”分组，每行提供读、写开关。
2. 可审核语言：多选具体语言，并与语言审核策略中的授权审核人保持双向同步。

交互规则：

- 勾选写权限自动勾选读权限。
- 取消读权限自动取消写权限。
- 超级管理员的矩阵全部选中并只读。
- 停用账号的整个抽屉只读。
- 保存时一次性更新页面权限和语言授权。

## 组件与数据流

新增集中式权限模块，包含：

- 权限资源注册表：唯一维护资源键、中文名称、分组和路由映射。
- 权限规范化函数：补齐旧本地数据缺失的权限字段。
- `canRead`、`canWrite`、`hasAnyLanguageReviewAccess` 等纯判断函数。
- 当前操作者权限 Hook/上下文，供导航、路由和页面按钮使用。
- 路由访问拦截组件和无权访问状态页。
- 写权限按钮包装组件，统一禁用状态和提示文案。

数据流为：人员权限抽屉修改矩阵 → Store 原子更新操作者权限 → 订阅组件重新渲染 → 导航、路由和页面操作立即使用新权限。

## 旧数据迁移

本地缓存中的旧操作者只有能力权限数组。加载时执行规范化：

- 超级管理员补齐全部页面读写权限。
- 普通操作者默认保留所有非超级管理员专用页面的读权限，避免升级后突然失去原有可见内容。
- 旧 `content.create`、`content.submit` 映射为人工消息任务和人工消息模板写权限。
- 旧 `variable.manage` 映射为模板变量写权限。
- 旧 `business.review`、`risk.review` 映射为审核中心写权限。
- 旧能力字段在迁移后不再作为界面授权依据。

## 异常与边界处理

- 当前操作者不存在或账号停用：所有后台受控页面显示无权访问。
- 权限资源缺失：按无权限处理，不采用默认放行。
- 系统配置没有任何可读子 Tab：隐藏入口；直接访问显示无权访问。
- 语言审核策略未配置授权人：沿用现有至少一名授权审核人的校验。
- 对无写权限按钮的程序化调用仍由统一写权限判断阻断，不能只依赖按钮样式。

## 验收口径

- 超级管理员能看到并读写全部页面，能编辑普通操作者权限。
- 普通操作者仅看到拥有读权限的导航项目。
- 无读权限页面通过直接 URL 访问时显示无权访问。
- 只读页面所有写操作置灰并显示权限提示。
- 写权限与读权限联动正确。
- 人工模板与事件模板可分别授权。
- 系统配置各子 Tab 可分别授权。
- 多语言审核只由语言授权决定，并限制到具体语言。
- 刷新页面后权限仍从本地 Store 恢复。

## 验证约束

本次按用户要求不执行测试。实现中仍保持权限判断为集中式纯逻辑，便于后续补充自动化测试和后端鉴权对接。
