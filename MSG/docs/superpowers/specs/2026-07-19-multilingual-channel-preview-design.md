# 多语言分渠道成品预览设计

## 背景

当前“多语言流程”中，每个目标语言都可以展开查看源文、机器翻译和人工校对字段，但不能按正式渠道查看最终消息效果。现有翻译数据也只有一套标题、摘要和正文，不能准确表达站内信与 App Push 原本不同的文案。

## 目标

- 每个语言行始终提供“预览”入口，不受翻译或审核状态限制。
- 预览所选正式渠道的成品效果：Web 站内信、App 站内信、App Push。
- 预览优先使用该语言的最新有效内容：审核通过稿、人工修订稿、机器翻译稿。
- 小语种专项审核只限制修改和通过操作，不限制预览。
- 兼容已有的单层翻译数据。

## 非目标

- 不修改业务审核、风控审核或小语种专项审核规则。
- 不在预览中提供内容编辑能力。
- 不为尚未返回翻译结果的语言生成虚假译文。

## 数据设计

翻译批次新增可选的源渠道内容与正式渠道：

- `channels`：本次内容实际使用的正式渠道。
- `sourceChannelContent`：创建翻译任务时冻结的完整站内信与 Push 源内容。

每个语言翻译项新增三层可选的分渠道内容：

- `machineChannelOutput`：外部机器翻译返回的分渠道结果。
- `humanChannelDraft`：人工修订的分渠道草稿。
- `approvedChannelOutput`：审核通过后冻结的分渠道结果。

每一层均按 `web` 和 `push` 保存内容。已有的 `machineOutput`、`humanDraft`、`approvedOutput` 字段继续保留作为旧数据兼容层。

内容解析顺序如下：

1. `approvedChannelOutput`
2. `humanChannelDraft`
3. `machineChannelOutput`
4. 旧数据中的审核通过稿、人工修订稿或机器翻译稿

旧数据回退时，将标题、摘要、正文映射到站内信；Push 使用标题和正文，并继承源内容中的图片、Deep Link、设备平台等非翻译配置。该回退仅用于历史数据展示，新任务必须保存分渠道翻译结果。

## 页面交互

每个语言行保留现有“查看译文”按钮，并新增“预览”按钮：

- “查看译文”：展示源文、机器翻译和人工校对内容，继续承担校对和普通语言审核操作。
- “预览”：在当前语言行下方展示成品预览，不跳转页面。
- 同一时间只展开一个语言的一种视图；点击另一语言或另一视图时直接切换。
- 已通过、翻译返回待审核、专项审核语言均可预览。
- 无翻译结果时，“预览”仍可点击，展开后明确显示“暂无可预览内容”。

成品预览复用现有 `MessagePreview`，根据批次记录的正式渠道只展示对应模块：

- 选择站内信：展示 Web 站内信与 App 站内信。
- 选择 Push：展示 App Push。
- 两者同时选择：展示全部三个预览。

## 数据流

1. 模板或临时消息提交外部翻译任务时，将正式渠道与完整源渠道内容写入翻译批次。
2. 原型中的外部翻译返回同时生成各渠道机器翻译结果。
3. 普通校对或专项审核保存时，同步更新分渠道人工稿或审核通过稿。
4. 用户点击某语言的“预览”，页面解析该语言最新有效内容并交给 `MessagePreview` 渲染。

## 异常与兼容

- 外部翻译无结果：展示空状态和已有错误信息，不使用源语言冒充译文。
- 缺少正式渠道记录：从关联模板或任务读取渠道；仍缺失时按现有内容判断可展示渠道。
- 缺少分渠道翻译结果：按旧字段回退，保证历史记录可预览。
- 缺少非翻译配置：使用现有 `MessagePreview` 的空状态文案。

## 测试范围

- 每个语言状态均展示“预览”按钮。
- 点击不同语言的“预览”可切换对应内容。
- 预览内容遵循“审核通过稿 → 人工稿 → 机翻稿”的优先级。
- 只渲染该任务选择的正式渠道。
- 小语种专项审核状态可以预览但不能在进度抽屉中编辑。
- 无结果语言展示空状态。
- 旧版单层翻译数据仍可预览。
