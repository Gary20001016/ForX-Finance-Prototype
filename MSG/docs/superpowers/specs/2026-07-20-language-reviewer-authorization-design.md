# 按语言指定审核人设计

## 背景

当前语言审核策略分别维护“审核组”“审核人数”和“允许提交人自审”。这些配置不能直接回答某个操作者是否有权审核某种语言，也会让审核人数与具体授权对象产生歧义。

本次将审核权限收敛为按语言直接授权具体操作者。

## 目标

- 每种语言直接配置一个或多个授权审核人。
- 取消审核组和审核人数概念。
- 被授权的审核人可以审核本人提交的内容。
- 普通语言当场校对和小语种集中审核使用同一套权限判断。

## 配置界面

语言审核策略表保留以下字段：

1. 语言：语言名称和区域代码。
2. 专项审核：决定该语言是否进入多语言审核页集中处理。
3. 授权审核人：从启用中的后台操作者里多选具体人员。
4. SLA：审核处理时限。
5. 超时动作：提醒、升级或阻断发布。
6. 状态：启用或停用该语言审核策略。

删除以下字段：

- 审核组。
- 审核人数。
- 允许提交人自审。

## 权限规则

- 每个启用的语言策略至少配置一名授权审核人。
- 当前操作者在目标语言的授权审核人列表中时，可以通过、驳回或保存该语言的审核修改。
- 当前操作者不在授权列表中时，可以查看审核内容，但所有会改变审核结果的操作均不可用，并显示“无该语言审核权限”。
- 授权审核人可以审核本人提交的内容，不做提交人与审核人隔离。
- 普通语言在模板、临时消息或事件内容页当场校对时执行相同权限判断。
- 专项审核语言仍进入“多语言审核”页，但不再按审核组筛选，改为默认只展示当前操作者有权审核的语言任务。

## 数据模型

`LanguageReviewPolicy` 使用 `authorizedReviewerIds: string[]` 保存人员标识。移除 `reviewGroup`、`reviewerCount` 和 `allowSubmitterReview`。

翻译任务继续保存目标语言，通过语言代码实时查找对应策略。策略变更只影响新建翻译任务的展示与权限快照；已有任务保留创建时的授权审核人快照，避免任务处理中权限范围无提示变化。

翻译条目增加 `authorizedReviewerIds?: string[]`，创建任务时从语言策略复制。历史数据没有该字段时，使用当前语言策略补齐。

## 操作者数据

前端原型提供统一的可授权操作者列表，至少包含操作者 ID、姓名、状态。授权选择器只展示启用中的操作者，并支持按姓名或 ID 搜索。

原型当前登录操作者继续使用现有页面传入的管理员标识；审核入口和抽屉基于该标识判断权限。

## 异常与边界

- 清空启用语言的全部授权审核人时阻止保存，并提示至少选择一人。
- 停用语言策略时允许暂时没有授权审核人。
- 已停用或不存在的操作者保留在已有策略中并标记为不可用，管理员必须重新选择后才能保存。
- 无权限人员直接访问审核抽屉时只能查看，不能提交审核结果。

## 前端改动范围

- 语言审核策略配置表及其布局样式。
- 语言策略和翻译条目的领域类型、种子数据与归一化逻辑。
- 普通语言当场校对的权限判断。
- 多语言审核列表的任务过滤、授权人展示和审核操作限制。
- 翻译审核抽屉的只读态与权限提示。

不修改后端接口，不新增审核人数、会签或角色组逻辑。

## 验收标准

- 管理员能为每种语言直接选择多个授权审核人。
- 页面不再显示审核组、审核人数和允许提交人自审。
- 授权人员能够审核该语言，包括本人提交的内容。
- 未授权人员只能查看，无法提交审核结果。
- 普通语言和专项审核语言执行一致的人员授权规则。
- 用户要求本次不运行自动化测试或构建校验。
