# 事件自动化 PRD

## 1. 模块摘要

事件自动化用于充值、提现、成交、强平、奖励和返佣等长期自动通知。系统拆为五类对象：事件目录、事件通知规则、规则内容版本、触发记录和渠道发送记录，不再使用“系统任务”这一混合概念。

## 2. 对象边界

| 对象 | 只负责什么 | 不负责什么 |
|---|---|---|
| 事件目录 | 描述发生了什么：编码、版本、调用方和 Schema | 不绑定模板、不保存发送状态 |
| 事件通知规则 | 描述是否通知、通知谁、用什么渠道和控制策略 | 不保存单次发送结果 |
| 规则内容版本 | 描述这次通知说什么及其多语言审核状态 | 不改变规则启停状态 |
| 触发记录 | 描述某事件实例命中某规则的一次执行 | 不作为长期配置 |
| 渠道发送记录 | 描述站内信或 Push 是否成功送达 | 不回写规则为“完成/失败” |

关系为：`事件定义 1—N 通知规则 1—N 内容版本`；运行时为：`事件实例 × 命中规则 → 触发记录 1—N 渠道发送记录`。

## 3. 必须支持的事件

| 事件编码 | 场景 | 默认分类 | 默认风险 | 关键变量 |
|---|---|---|---|---|
| `deposit.credited` | 充值到账 | 资产通知 | 重要 | 金额、币种、时间 |
| `withdrawal.succeeded` | 提现成功 | 资产通知 | 重要 | 金额、币种、地址、时间 |
| `withdrawal.failed` | 提现失败 | 资产通知 | 重要 | 金额、币种、失败原因、时间 |
| `order.filled` | 订单成交 | 交易通知 | 普通 | 交易对、方向、价格、数量、时间 |
| `liquidation.warning` | 强平预警 | 风控通知 | 紧急 | 交易对、保证金率、标记价格、时间 |
| `trial_fund.credited` | 体验金到账 | 奖励通知 | 普通 | 金额、币种、有效期 |
| `points.credited` | 积分到账 | 奖励通知 | 普通 | 积分值、来源、时间 |
| `commission.credited` | 返佣到账 | 奖励通知 | 普通 | 金额、币种、结算周期、时间 |

## 4. 主流程

```text
注册事件定义
└─校验并发布 Schema
   └─创建事件通知规则草稿
      └─提交审核→ 待审核
         ├─驳回→ 待修改
         └─通过→ 已启用
            └─接收事件实例
               └─去重、条件、主体、TTL 校验
                  ├─未命中/重复/过期→ 生成终态触发记录
                  └─命中→ 冻结当前内容版本
                     └─生成触发记录
                        └─拆分站内信与 App Push 渠道发送记录
```

单次触发完成、部分失败或失败后，规则仍保持`已启用`。只有操作者执行“停用规则”或规则过期，规则才停止接收新事件。

## 5. 不间断替换文字内容

```text
已启用规则（当前 V1）
└─创建内容版本→ V2 草稿；V1 继续生效
   └─提交机翻→ 机翻处理中
      └─外部任务回调→ 待人工审核
         └─人工审核通过→ 待审核
            └─业务审核通过→ 待生效
               └─发布版本→ V2 当前生效；V1 已替换
```

- 规则在全流程中保持`已启用`，不间断处理事件。
- 发布使用原子切换。切换前已进入处理的触发记录继续使用冻结的旧版本；切换后新触发使用新版本。
- 内容版本不可覆盖修改；发布错误时应创建回滚版本或将已审核旧版本重新发布，保留完整历史。

## 6. 功能需求

### 6.1 事件目录

- 事件编码全局唯一，发布后不可直接修改；Schema 变化使用新版本。
- Schema 描述字段名、类型、必填性、示例、敏感等级和说明。
- 必须提供`event_instance_id`、`subject_user_id`、`occurred_at`和`event_version`。
- 列表展示关联通知规则数和已启用数；详情提供 Schema 校验、创建通知规则和发送测试事件。

### 6.2 事件通知规则

- 一个事件可关联多条规则；每条规则独立配置条件、用户映射、渠道、TTL、频控、重试和负责人。
- 默认主体映射为`payload.user_id → UID`，不允许上传人工 UID 名单。
- 规则创建后先为草稿，审核通过并且存在当前生效内容版本后才能启用。
- 单条规则状态固定为：`草稿`、`待审核`、`待修改`、`已启用`、`已停用`、`已取消`、`已过期`。

| 当前状态 | 可执行操作 | 状态影响 |
|---|---|---|
| 草稿 | 查看详情、编辑规则、提交审核、取消规则 | 提交→待审核；取消→已取消 |
| 待审核 | 查看详情、撤回审核、取消规则 | 撤回→草稿；通过→已启用；驳回→待修改 |
| 待修改 | 查看详情、编辑规则、提交审核、取消规则 | 提交→待审核；取消→已取消 |
| 已启用 | 查看详情、创建内容版本、停用规则 | 创建内容版本不改变规则；停用→已停用 |
| 已停用 | 查看详情、编辑规则、启用规则、取消规则 | 编辑→草稿；启用→已启用；取消→已取消 |
| 已取消/已过期 | 查看详情 | 状态不变 |

### 6.3 规则内容版本与多语言

- 源语言为中文；操作者选择目标语言后，系统调用外部机翻任务接口。
- 外部接口异步回调后进入人工逐语言审核，全部语言通过后进入业务审批。
- 状态为：`草稿 → 机翻处理中 → 待人工审核 → 待审核 → 待生效 → 当前生效 → 已替换`；异常分支为`已取消`、`已回滚`。
- 每次触发必须冻结`content_version_id + template_version`，用于详情、审计和重放判断。

### 6.4 触发处理与幂等

- 幂等键固定为`rule_id:event_instance_id`，不得包含内容版本。
- 同一事件实例可命中多条规则，因此不同规则各生成一条触发记录。
- 同一规则收到重复事件实例时返回幂等成功，记录`重复抑制`，不重复创建用户消息。
- 条件不匹配、主体无效、事件过期也必须形成可查询的触发结果。
- 触发记录状态为：`已接收`、`处理中`、`已完成`、`部分失败`、`失败`、`已过滤`、`重复抑制`、`已过期`。

### 6.5 测试事件

- 测试表单按事件 Schema 生成字段，并限制到授权测试 UID 或测试环境。
- 测试结果展示命中规则、冻结内容版本、触发记录和关联渠道记录。
- 没有已启用规则或当前生效内容版本时明确阻断，不得伪造成功。

## 7. 字段定义

### 7.1 事件定义

`event_id`、`event_code`、`event_name`、`event_version`、`owner_team`、`caller`、`schema`、`status`、`created_at`、`updated_at`。

### 7.2 通知规则

| 字段 | 必填 | 说明 |
|---|---|---|
| `rule_id`、`rule_name` | 是 | 稳定规则标识和名称 |
| `event_id`、`event_version` | 是 | 关联事件和冻结 Schema 版本 |
| `condition_expression` | 是 | 安全条件表达式 |
| `subject_mapping` | 是 | 事件主体到用户 UID 的映射 |
| `channels` | 是 | 站内信、App Push |
| `current_content_version_id` | 启用时是 | 当前生效内容版本 |
| `dedupe_key_expression` | 是 | 固定生成`rule_id:event_instance_id` |
| `frequency_cap` | 是 | 频率限制 |
| `event_ttl_seconds` | 是 | 迟到事件有效期 |
| `max_retries`、`retry_backoff_seconds` | 是 | 临时失败重试策略 |
| `status`、`owner` | 是 | 规则状态与负责人 |

### 7.3 内容版本

`content_version_id`、`rule_id`、`version`、`template_id`、`template_version`、`source_locale`、`target_locales`、`localized_contents`、`translation_batch_id`、`status`、`created_by`、`created_at`、`activated_at`。

### 7.4 触发记录

`trigger_id`、`event_instance_id`、`event_id`、`rule_id`、`rule_version`、`content_version_id`、`template_version`、`idempotency_key`、`subject_user_id`、`status`、`received_at`、`completed_at`、`channel_total`、`success_count`、`failure_count`、`reason`、`trace_id`。

## 8. 权限与审计

- 事件 Schema、规则审核、规则启停、内容版本发布和测试事件使用独立权限。
- 规则创建人不能作为最终审核人；翻译审核人不能审核自己创建的源文案。
- 记录规则配置差异、内容版本、审批结论、原子切换、测试事件、重复抑制和失败原因。

## 9. 异常与边界

- 未知事件或版本、缺少实例 ID/主体/必填变量：拒绝并记录永久错误。
- 无已启用规则：记录未路由，不生成渠道发送。
- 无当前生效内容版本：规则不可启用或触发失败并告警。
- 外部机翻超时或失败：内容版本停留在机翻流程，现网当前版本继续生效。
- 新版本审核驳回：新版本回到修改流程，规则和当前版本不变。
- 发布请求超时：按版本 ID 查询发布结果，不重复切换。

## 10. 数据与指标

按事件、规则、内容版本和时间统计接收量、命中率、过滤率、重复率、过期率、成功率、部分失败率、处理时长和渠道失败分布。规则状态指标与触发结果指标分开展示。

## 11. 验收标准

1. 导航区分人工消息任务、事件通知规则、事件目录、触发记录和渠道发送记录。
2. 事件目录不直接绑定模板，只展示关联通知规则。
3. 已启用规则可创建新内容版本，并完成外部机翻、人工审核、业务审批和原子发布。
4. 内容发布期间规则持续启用；新旧触发分别冻结正确内容版本。
5. 幂等键不包含内容版本，同一事件在版本切换前后不会重复发送。
6. 单次触发结果进入触发记录，不改变规则为发送中、已完成或失败。
7. 每条触发记录可追踪到规则、内容版本和所有渠道发送记录。

## 12. 非本模块范围

业务事件总线、资金或交易状态机、营销旅程和跨事件编排不属于消息中心一期。
