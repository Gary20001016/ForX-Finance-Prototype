# 事件目录页面 PRD

> 文档范围：ForX Finance 消息中心后台「事件自动化 / 事件目录」页面  
> 页面路由：`/events`  
> 文档依据：当前前端页面和 Prototype Store 的实际实现  
> 文档状态：现状功能说明

## 1. 摘要

事件目录用于登记能够触发自动消息的上游业务事件，例如充值到账、提现成功、订单成交和强平预警。页面只维护事件本身的编码、版本、调用方和可用字段，不直接配置消息文案、发送渠道或目标用户；这些发送行为由后续的事件通知规则负责。

## 2. 联系人

| 角色 | 联系人 | 职责 |
|---|---|---|
| 产品负责人 | 待补充 | 确认事件目录边界、字段和状态规则 |
| 消息平台负责人 | 待补充 | 提供事件注册、联调和运行指标接口 |
| 上游业务研发 | 各事件调用方 | 提供事件编码、版本和字段定义 |
| 前端负责人 | 待补充 | 实现列表、注册弹窗、详情抽屉和权限控制 |
| QA | 待补充 | 验证注册、查询、规则跳转和测试事件流程 |

## 3. 背景

充值、提现、成交、风控和奖励等通知由业务系统持续触发。若消息平台直接在模板或发送任务中填写事件名称，容易产生以下问题：

- 同一事件被重复定义，编码和字段不一致。
- 模板不知道可以使用哪些事件字段。
- 无法区分“业务事件定义”和“消息发送规则”。
- 无法查看某个事件关联了多少条通知规则。
- 联调、测试和运行状态缺少统一入口。

因此系统增加事件目录，作为消息中心对业务事件的统一登记入口。

### 3.1 对象边界

| 对象 | 负责内容 | 不负责内容 |
|---|---|---|
| 事件目录 | 事件编码、名称、版本、业务线、调用方、字段和运行概况 | 不绑定模板，不配置渠道，不保存单次发送结果 |
| 事件通知规则 | 触发条件、用户映射、模板、渠道、去重和重试 | 不重新定义事件字段 |
| 触发记录 | 某个事件实例命中规则后的执行结果 | 不作为长期事件配置 |
| 渠道发送记录 | 站内信或 Push 的具体发送结果 | 不修改事件目录状态 |

## 4. 目标

### 4.1 产品目标

- 让有权限的操作者能够登记一个新的业务事件。
- 让操作者快速查询事件及其运行概况。
- 让操作者查看事件字段和关联通知规则。
- 让操作者从事件详情直接创建通知规则。
- 让操作者验证已有规则是否可以被测试事件正常触发。

### 4.2 成功指标

当前前端未接入正式统计接口，以下指标需在后端接入后计算：

| 指标 | 目标 |
|---|---|
| 事件编码重复率 | 0% |
| 已注册事件 Schema 完整率 | 100% |
| 待联调事件平均联调时间 | 可按业务线统计 |
| 测试事件成功率 | 可按事件和通知规则统计 |
| 事件到通知规则的配置成功率 | 可按创建漏斗统计 |

## 5. 目标用户

| 用户 | 主要任务 | 页面权限需求 |
|---|---|---|
| 消息平台管理员 | 管理事件定义，查看关联规则和运行状态 | 事件目录读、写权限 |
| 业务研发 | 确认调用方、版本和事件字段 | 事件目录读权限；是否开放写权限由管理员配置 |
| 消息运营 | 查看可用事件并创建通知规则 | 事件目录读权限、事件通知规则写权限 |
| QA / 联调人员 | 校验事件字段并发送测试事件 | 事件目录读、写权限 |
| 只读审计人员 | 查询事件和规则关联关系 | 事件目录读权限 |

## 6. 用户价值

- 操作者可以明确知道系统当前支持哪些业务事件。
- 创建通知规则时不需要手工猜测事件编码和字段。
- 事件定义和通知内容分开管理，同一个事件可以复用到多条通知规则。
- 可以从事件详情直接查看关联规则，减少跨页面查找。
- 测试事件会形成触发记录和渠道发送记录，便于后续排查。

## 7. 页面方案

### 7.1 页面结构

```text
事件目录
├─ 页面说明
├─ 注册事件
├─ 查询条件
│  ├─ 事件编码或名称
│  ├─ 业务线
│  ├─ 调用状态
│  └─ 重置
├─ 事件列表
├─ 事件详情抽屉
│  ├─ 基础信息
│  ├─ 事件字段 Schema
│  ├─ 关联通知规则
│  ├─ 校验 Schema
│  ├─ 创建通知规则
│  └─ 发送测试事件
└─ 注册业务事件弹窗
```

### 7.2 页面标题区

| 元素 | 页面内容 | 交互 |
|---|---|---|
| 页面标题 | 事件目录 | 无 |
| 页面说明 | 只维护上游业务事件的编码、版本与字段 Schema；通知内容和发送策略由事件通知规则管理 | 无 |
| 注册事件 | 主按钮 | 点击打开“注册业务事件”弹窗；无写权限时不可完成注册 |

### 7.3 查询区

| 查询项 | 类型 | 可选值/规则 | 实际行为 |
|---|---|---|---|
| 事件编码或名称 | 搜索框 | 自由输入 | 同时匹配事件编码和事件名称，不区分大小写 |
| 业务线 | 单选下拉 | 资产、交易、风控、奖励 | 精确筛选 `line` 字段 |
| 调用状态 | 单选下拉 | 运行正常、轻微延迟、待联调 | 精确筛选 `status` 字段 |
| 重置 | 操作 | — | 清空搜索词、业务线和调用状态 |

查询仅作用于前端当前已加载的数据，没有单独的“查询”提交步骤。

### 7.4 事件列表

| 列名 | 字段/展示规则 | 说明 |
|---|---|---|
| 事件 | 事件名称 + 事件编码 | 编码使用等宽样式展示 |
| 业务线 | `line` | 当前支持资产、交易、风控、奖励 |
| 版本 | `version` | 当前示例为语义化版本，如 `1.0.0` |
| 关联通知规则 | 规则总数 + 已启用数量 | 根据 `rule.eventId = event.id` 实时统计 |
| 调用方 | `caller` | 以标签形式展示上游服务名 |
| 近 24h 调用 | `calls` | 当前原型使用格式化字符串 |
| 失败率 | `failure` | 当前原型使用百分比字符串 |
| 最后调用 | `last` | 新注册事件显示“尚未调用” |
| 状态 | `status` | 当前展示运行正常、轻微延迟或待联调 |
| 操作 | 详情 | 打开当前事件的详情抽屉 |

### 7.5 注册业务事件

#### 7.5.1 表单字段

| 字段 | 必填 | 默认值 | 页面规则 | 注册后的用途 |
|---|---:|---|---|---|
| 事件编码 | 是 | 无 | 当前仅校验非空 | 作为事件主标识，如 `deposit.credited` |
| 事件名称 | 是 | 无 | 当前仅校验非空 | 用于列表和选择器展示 |
| 业务线 | 否 | 资产 | 下拉选择 | 用于分类和筛选 |
| 事件版本 | 否 | `1.0.0` | 自由输入 | 创建规则时冻结到规则配置 |
| 建议 TTL | 否 | 300 秒 | 最小值为 1 | 当前页面提交时未保存，详见 7.11 |
| 调用方 | 是 | 无 | 当前仅校验非空 | 标识产生该事件的上游服务 |
| 事件说明 | 否 | 无 | 多行文本 | 展示事件业务含义；当前详情未展示 |
| 事件字段 | 否 | 预置常用字段 | 使用英文逗号分隔 | 拆分、去空格、过滤空值后保存为字段列表 |

预置事件字段为：

```text
user_nickname, amount, currency, symbol, occurred_at
```

#### 7.5.2 注册结果

点击“注册并待联调”后，系统在事件列表头部新增一条记录，并自动设置：

| 字段 | 初始值 |
|---|---|
| 近 24h 调用 | `0` |
| 失败率 | `0%` |
| 最后调用 | 尚未调用 |
| 状态 | 待联调 |

注册成功后关闭弹窗、清空表单，并提示“事件已注册并进入待联调状态”。

### 7.6 事件详情

详情通过右侧抽屉展示，标题格式为“事件详情 · 事件名称”。

#### 7.6.1 基础信息

| 字段 | 展示内容 |
|---|---|
| 事件编码 | `id` |
| 版本 | `version` |
| 调用方 | `caller` |
| 业务线 | `line` |
| 变量数量 | `variables.length` |
| 最近测试 | `lastTestAt`，没有记录时显示“尚未测试” |

#### 7.6.2 事件字段 Schema

页面将事件字段按以下形式展示：

```text
{{ user_nickname }} · {{ amount }} · {{ currency }}
```

当前字段只包含字段名，不包含字段类型、是否必填、示例值或敏感等级。

### 7.7 关联通知规则

系统按事件编码查找所有 `eventId` 相同的事件通知规则。

| 列名 | 展示内容 |
|---|---|
| 通知规则 | 规则名称 + 规则 ID |
| 当前内容版本 | 当前生效内容版本的版本号；不存在时显示 `—` |
| 触发条件 | 条件表达式 |
| 渠道 | 站内信、Push 标签 |
| 状态 | 通知规则当前状态 |

若事件没有关联规则，页面显示：

> 尚无通知规则。该事件当前不会生成通知。请创建通知规则、发布首个内容版本并启用后再发送测试事件。

### 7.8 校验 Schema

点击“校验 Schema”时，当前前端直接提示“事件 Schema 与字段定义校验通过”。

该按钮目前属于交互占位，不会真正检查字段格式、必填字段、字段类型或版本兼容性。正式版本应接入后端 Schema 校验接口后才能作为真实校验结论。

### 7.9 创建通知规则

点击“创建通知规则”后跳转到事件通知规则页面，并携带当前事件编码：

```text
/automation
state.eventId = 当前事件 ID
```

目标页面应自动带入当前事件。按钮要求操作者同时拥有“事件通知规则”写权限；只有事件目录写权限但没有规则写权限时，不应允许创建规则。

### 7.10 发送测试事件

点击“发送测试事件”后，系统按以下顺序处理：

```text
确认事件存在
  → 查找该事件的一条已启用通知规则
  → 查找规则当前生效的内容版本
  → 按规则渠道生成测试发送记录
  → 生成一条测试触发记录
  → 更新事件最近测试时间
  → 增加规则近 24h 触发次数
```

#### 7.10.1 成功结果

- 站内信渠道生成一条站内信测试发送记录。
- Push 渠道生成一条 FCM Sandbox 测试发送记录。
- 测试用户固定为 `UID TEST-001`。
- 发送记录状态为“已送达”。
- 触发记录状态为“已完成”。
- 页面提示命中的规则 ID 和生成的触发记录 ID。

#### 7.10.2 失败结果

| 场景 | 页面提示 |
|---|---|
| 事件不存在 | 系统事件不存在 |
| 没有已启用通知规则 | 没有已启用的事件通知规则 |
| 规则没有当前生效内容版本 | 规则没有当前生效内容版本 |

当前测试事件不提供 Payload 编辑表单，不会按事件 Schema 要求操作者填写测试字段。

### 7.11 当前实现限制

以下内容必须明确标注为原型限制，不应对外描述为已经具备的正式能力：

1. 注册数据保存在前端 Prototype Store，并通过浏览器本地存储持久化，没有接入正式事件注册 API。
2. “建议 TTL”存在于注册表单，但提交事件时没有保存到事件定义中。
3. “校验 Schema”只显示成功提示，没有实际校验逻辑。
4. 事件编码没有重复校验，也没有格式校验。
5. 事件字段只保存名称，没有类型、必填性、说明、示例和敏感等级。
6. 没有正式的“待联调 → 运行正常”状态转换操作。
7. 待联调事件当前仍会出现在事件选择器中。
8. 近 24h 调用量、失败率和最后调用时间没有接入监控系统。
9. 测试事件使用固定用户和固定成功结果，不模拟上游真实 Payload 或渠道失败。
10. 当前只选取一条已启用规则进行测试；正式运行时，同一个事件可能命中多条规则。

### 7.12 权限要求

| 能力 | 所需权限 | 无权限行为 |
|---|---|---|
| 查看事件目录 | 事件目录读权限 | 不展示页面入口或阻止访问 |
| 注册事件 | 事件目录写权限 | 按钮不可执行 |
| 查看事件详情 | 事件目录读权限 | 不展示页面入口或阻止访问 |
| 校验 Schema | 事件目录写权限 | 按钮不可执行 |
| 发送测试事件 | 事件目录写权限 | 按钮不可执行 |
| 创建通知规则 | 事件通知规则写权限 | 按钮不可执行 |

## 8. 发布范围

### 8.1 当前原型已经覆盖

- 事件列表及前端查询。
- 注册事件弹窗。
- 事件详情及字段展示。
- 关联通知规则统计和列表。
- 从事件跳转创建通知规则。
- 测试事件的原型闭环。
- 页面级读写权限控制。

### 8.2 正式一期需要补齐

- 事件编码唯一性和命名格式校验。
- 结构化 JSON Schema。
- 正式事件注册、查询和联调接口。
- 状态转换和操作审计。
- 待联调事件的使用限制。
- 可编辑测试 Payload 和测试 UID 限制。
- 调用量、失败率和最近调用监控数据。

### 8.3 后续版本

- Schema 版本兼容性比较。
- 上游服务身份认证和事件签名配置。
- 事件字段敏感等级和脱敏策略。
- 按业务线配置事件负责人和告警人。
- 事件停用、废弃和迁移流程。
- 多规则命中测试及逐规则执行详情。

## 9. 页面验收标准

1. 操作者可以按事件编码、名称、业务线和状态筛选事件。
2. 点击重置后，所有查询条件恢复为空。
3. 有事件目录写权限的操作者可以打开注册弹窗并提交必填字段。
4. 注册成功后，新事件出现在列表头部，状态为“待联调”。
5. 事件字段按逗号正确拆分，并在详情中以变量形式展示。
6. 详情展示事件基础信息、变量数量和最近测试时间。
7. 列表和详情正确展示事件关联的通知规则及已启用数量。
8. 没有关联通知规则时，页面明确说明该事件不会生成通知。
9. 拥有事件通知规则写权限时，可以从详情跳转并带入当前事件。
10. 存在已启用规则和当前内容版本时，发送测试事件能够生成触发记录和渠道发送记录。
11. 测试条件不满足时，页面展示明确失败原因。
12. 无写权限的操作者只能查看，不能注册、校验或发送测试事件。
