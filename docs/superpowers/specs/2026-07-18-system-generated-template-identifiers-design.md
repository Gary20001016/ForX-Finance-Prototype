# 系统生成模板标识设计

## 背景

当前模板新建抽屉要求操作者填写“模板编码”，列表和详情又同时展示模板 ID 与模板编码。两者都是系统关联标识，操作者既无法判断命名规则，也可能因重名、格式错误或后续修改破坏任务和事件关联。

## 方案比较

### 方案 A：数据层生成 ID 和内部编码（采用）

保存模板时由数据层同时生成模板 ID 与内部编码。前端只向操作者展示“模板编号”，不展示内部编码，也不接受任何编码输入。

优点是唯一性规则集中、所有创建入口一致，并能继续兼容现有依赖 `code` 的任务和事件数据。缺点是数据层需要调整创建参数类型。

### 方案 B：模板抽屉在前端生成编码

实现简单，但其他创建入口仍可绕过规则，且并发创建时更容易产生冲突，因此不采用。

### 方案 C：删除内部编码，只保留 ID

模型最简洁，但现有事件定义、任务快照和历史模板使用 `code + version` 建立兼容关联，删除会扩大迁移范围，因此不采用。

## 最终设计

### 创建与编辑

- 新建模板时，操作者只填写模板名称等业务字段，不显示“模板编码”。
- `saveTemplate` 不再接受调用方传入 `code`，由数据层生成模板 ID 和内部编码。
- 模板 ID 格式为 `TPL-{场景}-{日期}-{四位序号}`，场景分别使用 `MAN`、`EVT`、`SHR`。
- 内部编码由模板 ID 派生为小写下划线格式，例如 `tpl_man_20260718_0001`，仅供程序关联。
- 编辑模板与产生新版本时沿用原模板 ID 和内部编码；二者都不可修改。
- 已有模板保留原 ID 和编码，不做历史数据迁移。

### 操作者界面

- 新建和编辑抽屉删除“模板编码”输入框，模板名称扩展为同一行更宽的输入区域。
- 模板列表名称下方展示 `template.id`，不再展示 `template.code`。
- 搜索提示改为“搜索模板编号或名称”；底层仍可兼容按历史内部编码搜索，但不向操作者强调。
- 已发布模板详情只显示一项“模板编号”，不再同时显示“模板 ID”和“模板编码”。

### 临时消息

- 临时消息为创建翻译批次而生成的隐藏模板也使用同一个数据层生成规则。
- 临时消息页面不再自行拼接 `temporary_*` 编码。

## 数据约束

- `MessageTemplate.code` 暂时保留，以兼容事件绑定、旧任务快照和历史数据。
- 新建模板调用参数从类型层排除 `id` 与 `code`。
- 更新模板调用参数从类型层排除 `id` 与 `code`，防止其他页面绕过前端修改标识。
- 同一天同一场景的序号根据当前模板集合中的同前缀数量递增，并跳过已存在的 ID。

## 测试范围

1. 新建模板抽屉不再出现模板编码输入框。
2. 只填写模板名称及业务内容即可保存，生成的 ID 和内部编码符合规则。
3. 模板列表展示模板编号，不展示内部编码，并使用新的搜索提示。
4. 已发布模板详情只显示“模板编号”。
5. 临时消息创建翻译批次时不传入自定义模板编码。
6. 编辑现有模板后 ID 与内部编码保持不变。

## 不在本次范围

- 不批量改写已有模板 ID 或内部编码。
- 不修改事件定义与历史任务的兼容匹配逻辑。
- 不增加操作者自定义编号前缀或编号重置配置。
