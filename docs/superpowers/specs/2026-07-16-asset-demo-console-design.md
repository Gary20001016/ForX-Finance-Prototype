# ForX 资产中心演示控制台设计

## 目标

把所有仅用于驱动原型状态的控件移出手机界面，让手机区域只呈现真实用户会看到的产品交互，同时保留完整的流程演示能力。

## 方案比较

1. **上下文感知的右侧控制台（采用）**：桌面端在手机右侧显示独立控制台，只展示当前页面可用的模拟动作。优点是操作与当前页面对应、不会遮挡手机、演示人员容易理解。
2. **悬浮调试按钮**：在页面角落打开浮层。空间占用小，但容易遮挡界面，也会让原型与调试工具的边界再次变模糊。
3. **页面底部控制区**：在手机下方集中展示全部模拟动作。实现简单，但需要频繁滚动，当前页面与控制动作的对应关系较弱。

## 产品与演示边界

### 保留在手机内

- 账户切换、充值与提现表单、复制地址、扫码、钱包 Deep Link。
- 地址簿、MFA、安全确认、资金记录筛选和账户价值交互。
- “填入示例地址”继续保留，因为用户选择 B；它代表输入辅助，不属于本次移出的状态机控制。
- 真实产品中的提交、确认、返回和重试入口。

### 移到控制台

- 地址充值的“收到一笔 500”触发。
- 钱包连接失效、钱包账户不一致模拟。
- 充值异常场景选择与处理阶段推进。
- 提现异常场景选择与处理阶段推进。
- 划转失败模拟与异步处理完成。

手机内不得再出现“模拟”“演示”“推进演示状态”等原型措辞。

## 布局

- 大屏采用三栏：产品说明、390px 手机、约 280px 演示控制台。
- 控制台使用与页面一致的深色、细描边和绿色状态色，但视觉层级弱于手机。
- 控制台顶部显示“演示控制台”和当前页面名称；主体按当前 route 渲染对应动作。
- 无专用动作的页面显示简短空状态，不渲染无关按钮。
- 800px 以下保持纯手机视图并隐藏控制台，确保移动端仍像真实 App。

## 交互与数据流

- `render()` 同时更新手机屏幕和控制台。
- 控制台复用现有状态迁移函数与事件语义，不建立第二套状态机。
- 原有模拟控件的选择器迁移到控制台，测试通过新容器定位，避免误判静态页面画廊。
- 终态约束保持不变：已入账充值和已完成提现不能被异常选择器改写；已完成划转不能重复记账。

## 可用性与异常处理

- 控制按钮随状态禁用，并在标题或辅助文字中解释不可操作原因。
- 控制台操作后手机立即刷新，当前 route 不发生意外跳转。
- 如果当前流程对象尚不存在，控制台显示进入对应产品流程的提示，不制造隐藏状态。
- 控制台自身不进入静态页面画廊，避免把演示工具误当成产品页面。

## 验收标准

- 手机内部没有原型专用模拟按钮、异常场景选择器或“完成演示处理”文案。
- 桌面控制台可以完成原有全部正常、异常和状态推进演示。
- 390×844 与 320px 宽度下控制台隐藏且手机无横向溢出。
- 充值、提现、划转的余额与终态幂等测试继续通过。
- 桌面截图能清楚识别手机产品与外部演示控制台的视觉边界。
