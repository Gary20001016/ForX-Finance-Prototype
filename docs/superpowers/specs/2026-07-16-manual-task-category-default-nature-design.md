# 人工消息任务继承分类默认消息性质设计

## 背景

当前人工消息任务创建页允许操作者再次选择“事务 / 服务 / 营销”，而系统配置的消息分类已经维护默认消息性质。两处都可配置会产生冲突，也无法保证同一分类的任务使用一致的消息性质。

## 目标

- 人工消息任务的消息性质统一由消息分类配置决定。
- 操作者能够看到系统自动确定的消息性质，但不能在任务中修改。
- 草稿、临时消息、多语言任务、预览和正式提交使用同一个消息性质。
- 历史任务保留创建或重新保存时冻结的消息性质，不随后续分类配置变化。

## 分类与默认性质

系统配置中的“消息分类”继续作为默认消息性质的唯一配置入口。当前默认映射为：

| 消息分类 | 默认消息性质 |
| --- | --- |
| 系统公告 | 服务 |
| 交易通知 | 事务 |
| 资产通知 | 事务 |
| 安全通知 | 事务 |
| 奖励通知 | 事务 |
| 活动通知 | 营销 |
| 风控通知 | 事务 |

本次前端原型使用共享的分类配置定义，避免系统配置页和任务创建页分别维护映射。

## 人工任务创建页

1. 操作者选择消息分类。
2. 页面立即读取该分类的默认消息性质。
3. 原有可操作的单选按钮改为只读展示：`消息性质：事务（由消息分类自动确定）`。
4. 切换消息分类时，只读值同步更新。
5. 编辑已有任务时，页面先展示任务已冻结的历史性质；如果操作者重新选择分类，则按新分类重新确定性质。

## 数据流

- 新建任务：`category -> category default nature -> task submission.nature`。
- 保存草稿：写入当前分类对应的默认性质。
- 临时消息与机翻批次：临时模板写入同一个默认性质。
- 预览摘要：展示由分类计算出的默认性质。
- 正式提交：冻结当前分类和默认性质，供任务列表、详情和审计使用。
- 历史任务：读取任务自身已保存的 `nature`，不会因分类配置后续变更而批量回写。

## 异常处理

- 分类缺失或未识别时，阻止进入下一步并提示重新选择有效分类，不静默使用“事务”兜底。
- 已停用分类后续接入持久化配置时，不允许创建或重新提交新任务；本次原型不扩大到分类停用逻辑。

## 测试范围

- 人工任务创建页不再出现可编辑的消息性质单选按钮。
- 默认“系统公告”展示“服务”。
- 切换到“活动通知”后展示“营销”。
- 保存或提交的数据使用分类默认性质。
- 临时消息创建翻译任务时使用同一个默认性质。
- 任务预览展示自动确定的消息性质。

## 非目标

- 不修改风险等级的现有选择逻辑。
- 不批量重写历史任务数据。
- 不在本次改造中实现分类配置的后端持久化接口。
