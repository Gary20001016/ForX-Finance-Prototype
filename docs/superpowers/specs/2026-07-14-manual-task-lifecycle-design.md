# 人工消息任务状态简化设计

## 1. 目标

统一人工消息任务的状态、操作名称和操作结果，消除“已通过”“部分失败”“失败”等审核或发送结果与任务生命周期混用的问题。系统事件触发任务继续使用独立的启用状态模型，不纳入本次改造。

## 2. 数据模型

人工任务拆分为三个独立字段：

- `taskStatus`：`草稿`、`待审核`、`待修改`、`待发送`、`发送中`、`已暂停`、`已完成`、`已取消`、`已过期`。
- `approvalStatus`：`未提交`、`审核中`、`通过`、`驳回`、`已撤回`、`已终止`。
- `deliveryResult`：`未开始`、`处理中`、`成功`、`部分失败`、`失败`。

其中 `deliveryResult` 不再作为任务状态。人工任务执行结束后统一进入 `已完成`，通过发送结果说明最终质量。

## 3. 主流程

```text
草稿
└──提交审核→ 待审核
    ├──撤回审核→ 草稿
    ├──驳回审核→ 待修改
    │   ├──编辑任务→ 待修改
    │   └──提交审核→ 待审核
    └──通过审核
        ├──[定时发送]→ 待发送
        │   ├──系统启动发送→ 发送中
        │   ├──取消任务→ 已取消
        │   └──系统标记过期→ 已过期
        └──[立即发送]→ 发送中
            ├──暂停发送→ 已暂停
            │   ├──恢复发送→ 发送中
            │   ├──取消任务→ 已取消
            │   └──系统标记过期→ 已过期
            ├──取消任务→ 已取消
            └──系统完成发送→ 已完成
                ├──发送结果：成功
                ├──发送结果：部分失败
                └──发送结果：失败
```

## 4. 操作名称

用户操作统一使用：`查看详情`、`编辑任务`、`复制任务`、`提交审核`、`撤回审核`、`通过审核`、`驳回审核`、`取消任务`、`暂停发送`、`恢复发送`、`重试失败项`。

系统动作统一使用：`系统启动发送`、`系统完成发送`、`系统终止发送`、`系统标记过期`。计划时间和有效期是触发条件，不作为操作名称。

## 5. 前端行为

### 5.1 列表与筛选

- 人工任务状态筛选只展示九个标准状态，不再出现 `已通过`、`已驳回`、`部分完成`、`部分失败`或`失败`等混合状态。
- 列表分别展示“任务状态”“审核状态”“发送结果”。未开始发送时，发送结果显示 `未开始`；发送中或已暂停显示 `处理中`。
- `已完成 + 部分失败`和`已完成 + 失败`均保持任务状态为 `已完成`。

### 5.2 操作菜单

- 所有状态支持 `查看详情`；所有任务支持 `复制任务`，原任务状态不变，新任务从草稿开始。
- `编辑任务`仅对 `草稿`、`待修改`、`待发送`开放。编辑待发送任务时提示原审批结果将失效，保存后回到草稿。
- `待审核`支持 `撤回审核`，撤回后回到草稿。
- `发送中`支持 `暂停发送`和`取消任务`。
- `已暂停`支持 `恢复发送`和`取消任务`，不支持直接编辑。
- `已完成`且发送结果为 `部分失败`或`失败`时支持 `重试失败项`。原任务保持已完成，前端打开新的重试任务草稿。
- `已取消`和`已过期`为不可变终态，只支持查看和复制。

### 5.3 审核动作

- 通过审核时，定时任务进入待发送；立即任务直接进入发送中。
- 驳回审核时，任务进入待修改，审核状态为驳回。
- 撤回审核时，任务进入草稿，审核状态为已撤回。

## 6. 系统事件任务边界

系统事件触发任务继续使用 `草稿 → 待审核 → 已启用 → 已停用/已过期`。单次事件的发送中、成功和失败只进入发送记录，不改变事件任务本身状态。

## 7. 验收标准

1. 人工任务列表、筛选、标签和详情仅使用九个标准任务状态。
2. 审核状态与发送结果可独立识别，不再把部分失败或失败显示为任务状态。
3. 操作菜单名称与本设计一致，并根据当前状态正确启用或禁用。
4. 每个用户操作产生符合主流程的状态变化。
5. 事件触发任务不被人工任务状态改造影响。
6. 现有任务创建、审核、暂停、恢复、取消、复制和重试交互具有自动化回归测试。
