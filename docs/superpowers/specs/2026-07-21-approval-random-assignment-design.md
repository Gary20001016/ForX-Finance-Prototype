# 审核工单随机指派设计

## 目标

审核工单提交后只指派给一名合格审核人。发起者不能审核自己的内容；审核人失去资格时，未完成工单立即改派，系统中不产生无人负责的待审核工单。

## 资格规则

审核人必须同时满足：

- 账号处于启用状态；
- 拥有“审核中心”页面写权限；
- 不是该工单的发起者。

语言审核权限独立配置，不参与发布审核工单的候选人筛选。

## 数据模型

每张审核工单保存唯一的 `assigneeId` 和审核人展示名称 `assignee`。工单不使用审核组、多级节点或多人会签。

## 指派流程

1. 提交任务或模板审核时，系统从合格审核人集合中随机选择一人。
2. 选中后将审核人固化在工单上，“待我审核”仅展示指派给当前操作者的待处理工单。
3. 如果候选集合为空，拒绝提交并提示联系超级管理员配置审核中心写权限。
4. 同一工单在审核完成前始终只有一个审核人。

## 权限变化与改派

超级管理员保存人员权限时，系统检查该人员名下所有待处理工单。如果该人员失去审核中心写权限或账号不再启用，则逐张从其余合格人员中随机改派。

权限更新与工单改派作为一次原子操作完成。如果任一待处理工单找不到替代审核人，则拒绝本次权限变更，原权限和原指派保持不变。

## 审核操作校验

通过或驳回前同时校验：

- 当前操作者仍拥有审核中心写权限；
- 当前操作者是工单的 `assigneeId`；
- 当前操作者不是 `submitterId`；
- 工单仍处于待处理状态。

不满足条件时只允许查看详情，不允许提交审核结果。前端负责按钮状态和提示，正式接口仍需执行相同校验。

## 页面表现

- “待我审核”：仅显示指派给当前操作者的待处理工单。
- “全部工单”：显示当前操作者有权查看的全部审核记录。
- 列表和详情展示唯一审核人。
- 非指派审核人进入详情时显示只读提示。
- 发起者即使拥有写权限也不能审核自己的工单。

## 异常处理

- 无可用审核人：阻止提交。
- 权限变化后无可接手人员：阻止权限变更。
- 审核过程中指派已变化：拒绝旧审核人的操作，并要求刷新工单。
- 重复提交审核结果：以后端工单状态和审核请求幂等键为准，只接受首次有效结果。

## 验收标准

- 新工单只产生一个非发起者审核人。
- 多次创建工单时可在合格审核人之间随机分布。
- “待我审核”不显示指派给其他人的工单。
- 非指派人、无写权限人员和发起者均无法通过或驳回。
- 审核人失去写权限后，其待处理工单全部改派。
- 无替代审核人时，权限变更不会生效。
