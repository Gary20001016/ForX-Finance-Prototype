# 消息中心按工作流分组导航设计

## 1. 背景

当前侧栏将人工任务、事件规则、模板、事件目录、触发记录和公共治理能力平铺展示。菜单同时混合“业务流程”“资源类型”和“运行结果”三种分类方式，操作者需要先理解系统对象关系，才能判断入口。

本次调整将导航改为工作流优先：人工运营和事件自动化分别形成连续工作区，多语言、审批、发送记录和分析作为共享治理能力统一收口。

## 2. 目标

- 操作者先按“人工发送”或“事件自动化”选择工作区。
- 人工任务、人工模板和人工受众保持相邻。
- 事件规则、事件模板、事件目录和触发记录保持相邻。
- 两个模板入口复用同一模板库、版本、多语言、审核和预览能力。
- 保留现有页面路径兼容性，不使已有收藏链接失效。
- 收起侧栏时仍能通过图标、提示和选中状态判断当前工作区。

## 3. 导航信息架构

```text
工作台

人工消息
├─ 人工消息任务
├─ 人工消息模板
└─ 用户与受众

事件自动化
├─ 事件通知规则
├─ 事件消息模板
├─ 事件目录
└─ 触发记录

运营与治理
├─ 多语言审核
├─ 审核中心
├─ 发送记录
└─ 数据分析

系统配置
```

分组默认展开。刷新页面后，根据当前路径自动展开所属分组；侧栏收起时仅显示一级分组图标，通过浮层展示组内入口。

## 4. 模板入口与数据边界

### 4.1 共享底层

“人工消息模板”和“事件消息模板”不是两套模板系统。二者共享：

- 模板实体和模板 ID；
- 内容版本；
- Web/App 站内信与 App Push 文案；
- 模板变量校验；
- 外部机器翻译、多语言校对和专项语言审核；
- 业务与风控审核；
- 预览、发布状态和审计记录。

### 4.2 适用场景

模板增加 `usageScope` 字段：

| 值 | 含义 | 可见入口 |
|---|---|---|
| `manual` | 仅用于人工消息任务 | 人工消息模板 |
| `event` | 仅用于事件通知规则 | 事件消息模板 |
| `shared` | 两类流程均可使用 | 两个入口 |

历史模板迁移时，先根据已有人工任务和事件规则引用关系推断：只被人工任务引用的设为 `manual`，只被事件规则引用的设为 `event`，两类均引用或无法判断的设为 `shared`。

### 4.3 页面行为

- 人工入口打开统一模板页，并应用 `usageScope in [manual, shared]`。
- 事件入口打开统一模板页，并应用 `usageScope in [event, shared]`。
- 页面标题、说明、新建按钮和空状态文案随入口场景变化。
- 从人工入口新建默认选择“人工消息”；从事件入口新建默认选择“事件通知”。操作者可改为“通用”，不能直接切换成另一个单一场景。
- 编辑“通用”模板时，必须同时通过人工变量和事件变量的适用性检查。
- 任务选择模板时只返回 `manual/shared`；事件规则选择内容模板时只返回 `event/shared`。

## 5. 路由与导航状态

保留现有 `/templates` 路径，通过查询参数表达入口：

- `/templates?scope=manual`：人工消息模板；
- `/templates?scope=event`：事件消息模板；
- `/templates`：兼容旧链接，默认进入人工消息模板视图。

模板详情、新建和编辑在返回列表时保留 `scope`。侧栏选中状态以路径和查询参数共同判断，避免两个模板入口同时高亮。

其他既有路径不变：

| 工作区 | 页面 | 路径 |
|---|---|---|
| 人工消息 | 人工消息任务 | `/tasks` |
| 人工消息 | 用户与受众 | `/segments` |
| 事件自动化 | 事件通知规则 | `/automation` |
| 事件自动化 | 事件目录 | `/events` |
| 事件自动化 | 触发记录 | `/triggers` |
| 运营与治理 | 多语言审核 | `/multilingual-review` |
| 运营与治理 | 审核中心 | `/approvals` |
| 运营与治理 | 发送记录 | `/deliveries` |
| 运营与治理 | 数据分析 | `/analytics` |
| 系统 | 系统配置 | `/settings` |

## 6. 面包屑与页面上下文

面包屑展示完整工作区层级：

- `消息中心 / 人工消息 / 人工消息模板`；
- `消息中心 / 事件自动化 / 事件消息模板`；
- `消息中心 / 运营与治理 / 多语言审核`。

从多语言审核、审核中心或发送记录进入对象详情时，详情页同时显示来源类型和来源对象，并提供“返回人工工作区”或“返回事件工作区”的明确入口。

## 7. 权限与可见性

- 菜单组只有在用户至少拥有一个子页面权限时显示。
- 子页面按原有权限独立隐藏，不因同组页面权限而自动开放。
- 同一模板的权限按动作控制，不按入口复制权限。
- 没有事件自动化权限的人工运营人员不会看到事件分组，但仍可在公共治理页面处理被授权的待办。

## 8. 异常与兼容处理

- 查询参数无效时回退人工模板视图，并保留页面可用性。
- 模板缺少 `usageScope` 时按历史迁移规则推断；无法推断时按 `shared` 展示。
- 用户直接打开无权限场景的模板链接时，显示无权限状态，不自动跳到另一入口。
- 公共治理页面不复制为人工版和事件版，使用来源筛选区分对象。

## 9. 验收标准

1. 侧栏按四个区域展示，人工和事件相关入口不再平铺混排。
2. 当前路径可正确展开分组、选中子菜单并生成三级面包屑。
3. 两个模板入口访问同一模板组件，但列表结果、标题和新建默认场景不同。
4. `shared` 模板在两个入口均可见，`manual` 和 `event` 模板不会跨入口出现。
5. 人工任务不能选择事件专用模板，事件规则不能选择人工专用模板。
6. `/templates` 旧链接仍能打开，并默认进入人工模板视图。
7. 多语言审核、审核中心、发送记录和数据分析仍能查看两类来源，并支持来源筛选。
8. 桌面展开、桌面收起和窄屏自动收起三种状态下均可完成导航。
9. 现有页面直接链接和浏览器刷新不丢失当前工作区。

## 10. 测试范围

- 导航配置单元测试：分组、顺序、路径与模板查询参数。
- 布局交互测试：展开、收起、自动展开、选中态和面包屑。
- 模板列表测试：人工、事件、通用三类过滤与新建默认值。
- 使用方测试：人工任务与事件规则的模板候选范围。
- 兼容测试：旧模板数据、无效查询参数和旧 `/templates` 链接。
- 回归测试：多语言审核、审批、发送记录和分析页面路由不变。

## 11. 非本次范围

- 不拆分模板数据库或复制模板版本。
- 不改变人工任务和事件通知规则的生命周期。
- 不新增独立的人工审批中心或事件审批中心。
- 不改变用户端站内信和 App Push 展示。
