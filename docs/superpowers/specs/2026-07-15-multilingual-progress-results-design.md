# 多语言进度内翻译结果查看与校对设计

## 目标

让运营人员在任务、模板等列表打开“查看多语言进度”后，无需离开当前业务上下文即可逐语言查看源文、机器翻译和人工修订结果。普通语言允许当场校对；配置为小语种专项审核的语言只能在独立“多语言审核”页面修改和审核。

## 页面结构

沿用现有 `MultilingualProgressDrawer`，保留批次汇总、流程步骤和逐语言列表。每个语言行增加“查看译文/收起译文”操作，展开区域显示：

1. 源语言和目标语言。
2. 源文标题、摘要、正文。
3. 机器翻译标题、摘要、正文。
4. 当前人工修订稿或最终审核稿。
5. 变量检查、翻译时间、审核人和审核时间。

同一时间只展开一个语言，避免多个长文本同时占用抽屉空间。

## 内容取值

- 源文优先读取翻译批次 `sourceContent`，旧数据缺失时回退到模板 Web 内容。
- 机器翻译优先读取 `machineOutput`，兼容回退到 `machineTitle`、`machineSummary`、`machineBody`。
- 当前结果按 `approvedOutput → reviewed* → humanDraft → machineOutput → machine*` 的顺序展示。
- `排队中`、`翻译中`且没有机器翻译内容时显示“翻译结果尚未返回”。
- `翻译失败`且没有结果时显示错误码、错误原因和“暂无可查看的译文”。

## 普通语言操作

普通语言仅在 `待普通确认`、`待人工审核`、`修改中`、`审核驳回`等可处理状态显示编辑表单：

- 操作者可修改标题、摘要和正文。
- “保存修订”写入人工草稿，但不改变为审核通过。
- “修订并通过”执行普通语言确认，更新语言状态和批次进度。
- 模板变量检查失败时禁止通过。
- 已通过语言只读展示最终审核结果，不能在进度抽屉中覆盖历史结果。

## 小语种专项审核边界

- `specialReviewRequired=true` 的语言在进度抽屉内始终只读。
- 展开后可查看源文、机翻和当前修订结果，但不显示输入框、保存、驳回或通过按钮。
- `待小语种专审`和`专审中`提供“前往多语言审核”按钮，跳转到 `/multilingual-review?item=<item_id>`。
- 任何进度抽屉操作都不得直接调用小语种专项审核通过接口。

## 组件与数据流

- `MultilingualProgressDrawer` 管理当前展开语言和普通语言编辑草稿。
- 复用现有 `saveTranslationDraft`、`approveOrdinaryTranslation` 状态操作，不新增数据模型。
- 每次保存或通过后从原型 Store 读取最新批次，使语言状态、内容和总进度立即刷新。
- 现有独立 `TranslationReviewDrawer` 和 `MultilingualReviewPage` 保持小语种专项审核职责。

## 兼容与非范围

- 不修改外部机器翻译任务接口和回调模型。
- 不修改需要专项审核的语言配置。
- 不修改人工消息、事件规则和模板的发布门禁。
- 不允许在进度抽屉中重新编辑已经通过的译文。

## 验收标准

1. 点击任一已返回结果的语言可直接查看源文、机翻和当前结果。
2. 普通待确认语言可保存修订并审核通过，批次进度随之更新。
3. 小语种专审语言只读，且只能跳转独立多语言审核页面处理。
4. 排队、翻译中和失败语言显示正确的空结果或错误状态。
5. 旧批次仅含 `machineTitle` 等字段时仍能查看译文。
6. 多语言相关聚焦测试、完整测试和生产构建通过。
